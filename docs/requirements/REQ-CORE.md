# REQ-CORE — コアエンジン

## REQ-CORE-001: ノードグラフトップレベルアーキテクチャ + Composition/Layer モデル

- **優先度**: Must
- **ステータス**: Revised (v3 — レイヤーネットワークモデル。詳細は REQ-LAYER)
- **説明**: Ravelの内部表現はノードグラフ（DAG）を基盤とする。Composition は独自の解像度・FPS・尺を持ち、Layer の順序付きリストを保持する。タイムラインUIは選択中の Composition 内の Layer 群を表示する「ビュー」として機能する。全てのデータフロー、エフェクト適用、合成はDAG上のノード接続で表現される。
- **Compositionモデル（AEモデル）**:
  - Composition は独自の解像度・FPS・尺を持つ（AEプリコンプ相当）
  - Layer は下から上への合成順序を持つ（AE標準）
  - **v3: 各 Layer は「殻」（ビルトインTransform、opacity、blend_mode、時間配置、親子付け等の汎用プロパティ）+「1つのノードネットワーク」で構成される（1レイヤー = 1ネットワーク、Houdini 的。REQ-LAYER-001）**
  - **v3: 従来の LayerSource 構造分岐とエフェクトサブグラフの二重構造は廃止。レイヤー種は作成時テンプレートに降格（REQ-LAYER-008）**
  - Layer の親子関係（Parenting）をサポート。Null Layer による制御。Transform継承（Null Layer は REQ-LAYER-005 で再定義）
- **評価方式（v3）**: 殻の合成チェーン（時間変換 → Transform → Opacity → Merge）は従来通り synthetic ノードへコンパイルするが、**v2 の「Composition 全体を平坦な DAG ノード列に展開し Evaluator 変更不要」方式は撤回**。レイヤーネットワークは平坦化せず、ネットワーク境界ノードを介した再帰評価とし、Evaluator は Document-aware 化・スコープ付き EvalContext・評価時パラメータ解決の変更を受け入れる（REQ-LAYER-006, REQ-LAYER-007）。殻の決定論的 ID による dirty 追跡・キャッシュ維持は継続して機能する。
- **受入条件**:
  - [ ] ルートノードグラフが存在し、全処理がDAG上のノードとして表現される
  - [ ] Composition が Layer の順序付きリストを保持する
  - [ ] 殻の合成チェーンが synthetic ノード列にコンパイルできる
  - [ ] タイムラインUIが Composition 内の Layer の in/out + キーフレームを表示する
  - [ ] Layer のノードネットワークをノードエディタで開いて編集できる（REQ-LAYER-011）
  - [ ] ノードグラフの変更がタイムライン表示に反映される
  - [ ] PreComp（入れ子 Composition）が動作する

## REQ-CORE-002: Hybrid Pull評価エンジン

- **優先度**: Must
- **ステータス**: Draft
- **説明**: ノードグラフの評価はPull型（遅延評価）を基盤とし、上流ノードの変更をPush的に通知してキャッシュを選択的に破棄するHybridモデル。出力ノードから逆方向に再帰的にデータを要求し、dirtyフラグが立ったノードだけを再評価する。
- **受入条件**:
  - [ ] 出力ノードからPull型で評価が実行される
  - [ ] パラメータ変更時にdirtyフラグが下流に伝播する
  - [ ] dirtyでないノードはキャッシュから結果を返す
  - [ ] 不要なノード（出力に接続されていないノード）は評価されない
  - [ ] トポロジカルソートに基づく評価順序が保証される
- **依存**: REQ-CORE-003, REQ-CORE-006

## REQ-CORE-003: 階層型トレイトベース型システム

- **優先度**: Must
- **ステータス**: Draft
- **説明**: ノード間を流れるデータの型をRustのトレイトシステムで定義。Buffer、Temporal、Geometric等のトレイトで接続可能性を判定。具体型としてフレームバッファ（RGBA/float）、クリップ、2Dシェイプ/パス/マスク、スカラー/ベクトル/カラー/カーブ、PCMバッファ、文字列、ポイントクラウド/パーティクル群等をサポート。
- **受入条件**:
  - [ ] トレイト境界でノード接続の型互換性がコンパイル時に検証される
  - [ ] 画像系、時間系、ジオメトリ系、数値系、オーディオ系、テキスト系、構造体系の型が定義されている
  - [ ] 型変換（暗黙/明示）のルールが定義されている
  - [ ] 新しい型の追加がトレイト実装で拡張可能

## REQ-CORE-004: イミュータブルデータ構造によるアンドゥ

- **優先度**: Must
- **ステータス**: Draft
- **説明**: グラフ全体をイミュータブルなpersistent data structureで保持。変更時は変更部分だけ新規作成し、残りは構造共有（`Arc<Node>`）。アンドゥは過去バージョンへのポインタ切り替え。`im`クレート活用を検討。
- **受入条件**:
  - [ ] グラフの変更が非破壊的に新バージョンを生成する
  - [ ] 未変更サブツリーがゼロコストで共有される
  - [ ] アンドゥ/リドゥが過去バージョンの参照切り替えで実現される
  - [ ] アンドゥ履歴のメモリ消費が構造共有により抑制される
- **依存**: REQ-CORE-001

## REQ-CORE-005: 専用スレッドプール + Tokio I/O

- **優先度**: Must
- **ステータス**: Draft
- **説明**: 用途別にスレッドを分離。UIスレッド（GPUI）、評価スレッドプール（rayon work-stealing）、GPUコマンド投入スレッド、デコードスレッドプール（FFmpeg）、オーディオスレッド（リアルタイム優先度）、Tokioランタイム（ファイルI/O、ネットワーク、プラグインホスト制御）。スレッド間はロックフリーキュー（crossbeam-channel等）で通信。
- **受入条件**:
  - [ ] UIスレッドがブロックされない
  - [ ] オーディオスレッドがリアルタイム優先度で動作する
  - [ ] 評価プールがrayonのwork-stealingで並列化される
  - [ ] スレッド間通信がロックフリーキューで行われる
  - [ ] TokioランタイムがI/O処理専用として分離されている

## REQ-CORE-006: 三層キャッシュ (VRAM/RAM/Disk)

- **優先度**: Must
- **ステータス**: Draft
- **説明**: ノード評価結果を三層でキャッシュ。GPU VRAM（最速、容量小）→ RAMメモリ（中速、中容量、LRU）→ ディスク（低速、大容量）と段階的にエビクト。プレビュー再生時はVRAMキャッシュから直接表示しゼロコピー。ディスクキャッシュは`.ravprj`内の`.cache/`に格納し、zip化時に除外可能。
- **受入条件**:
  - [ ] 三層キャッシュが動作しVRAMヒット時にGPU→CPU転送が発生しない
  - [ ] LRUベースのエビクションが各層で機能する
  - [ ] ディスクキャッシュが`.ravprj/.cache/`に永続化される
  - [ ] キャッシュヒット率がモニタリング可能
- **依存**: REQ-GPU-001, REQ-CORE-002

## REQ-CORE-008: マルチコンポジション + PreComp + サブグラフ共有

- **優先度**: Should
- **ステータス**: Revised (v2)
- **説明**: プロジェクト内に複数の Composition を作成可能。Composition を別 Composition の Layer ソース（PreComp）としてネスト可能。各 Composition は独自の解像度・FPS・尺を持つ。複数の CompNode から同一 Composition を参照可能（インスタンス共有）。サブグラフ（エフェクトチェーン等）の共有参照も可能。Composition はドキュメント層に `im::HashMap<CompId, Arc<Composition>>` として保持し、undo 対応。
- **受入条件**:
  - [ ] 複数の Composition をプロジェクト内に作成できる
  - [ ] Composition を別 Composition の Layer ソース（PreComp）としてネストできる
  - [ ] PreComp の循環参照（A⊂B⊂A）が編集時に検出・拒否される
  - [ ] 共有 Composition の変更が全参照元 CompNode に反映される
  - [ ] 各 Composition が独自の解像度・FPS・尺を持つ
- **依存**: REQ-CORE-001

## REQ-CORE-010: ジオメトリ属性システム

- **優先度**: Must
- **ステータス**: Draft
- **説明**: ジオメトリ（ポイント群、パス/カーブ、メッシュ、インスタンス群）をノード間を流れる第一級データ型として定義し、各要素に**任意名の属性**（attribute）を付与できる仕組みを提供する。属性はドメイン（point / primitive / instance / geometry 全体）ごとに保持され、型（float / vec2 / vec3 / vec4 / color / int / bool / string）を持つ。位置（`P`）、インデックス（`index`）、法線・スケール・回転・色等の標準属性名を予約する。下流ノードは属性を読み書き・追加・削除でき、複製やジオメトリ変換を通じて属性が伝播する。属性の転送（transfer: 近傍ジオメトリからのコピー）・昇格（promote: ドメイン間変換）・サンプリング（パス上の位置での補間取得）を標準操作として提供する。Geometry Nodes / Houdini 的プロシージャル制御の基盤。
- **受入条件**:
  - [ ] ポイント群/パス/インスタンス群のジオメトリ型がノード間を流れる
  - [ ] 任意名・任意型の属性を point / instance ドメインに付与できる
  - [ ] 標準属性（P/index 等）が全ジオメトリ生成ノードで設定される
  - [ ] 複製・変形・変換ノードを通過しても属性が保持・伝播される
  - [ ] 属性の転送・昇格・サンプリング操作がノードとして提供される
  - [ ] Lua 式から属性値を参照できる（`@attr` 相当のアクセス）
  - [ ] GeometricData トレイト（REQ-CORE-003）配下に統合される
- **依存**: REQ-CORE-003, REQ-CORE-004

## REQ-CORE-011: ステートフル評価とシミュレーションキャッシュ

- **優先度**: Must
- **ステータス**: Draft
- **説明**: パーティクル・物理・トレイル等、**前フレームの結果に依存するノード**を Hybrid Pull 評価（REQ-CORE-002）と整合する形でサポートする。ステートフルノードは「フレーム逐次評価」を宣言し、評価エンジンはフレーム t の要求に対し未計算区間 [last_cached+1, t] を順に評価してシミュレーションキャッシュに蓄積する。上流パラメータ/グラフ変更時はキャッシュを影響フレーム以降で破棄し再計算する。タイムラインスクラブ時は最後にキャッシュされたフレームまでの結果を表示し、再計算をバックグラウンドで進行する。同一シード・同一入力での評価は決定的であること。
- **受入条件**:
  - [ ] ステートフルノードがフレーム逐次で評価され前フレーム状態を参照できる
  - [ ] シミュレーション結果がフレーム範囲でキャッシュされる
  - [ ] 上流変更でキャッシュが影響フレーム以降のみ破棄される
  - [ ] 後方スクラブ（t が既キャッシュ範囲内）が再計算なしで表示される
  - [ ] 前方ジャンプ時に未計算区間が順次評価される（UI は非ブロック、REQ-CORE-005）
  - [ ] 同一シードで再現可能（決定的評価）
  - [ ] ステートレスノードの評価パス（REQ-CORE-002）に性能退行がない
- **依存**: REQ-CORE-002, REQ-CORE-006

## REQ-CORE-012: 汎用フィールド評価

- **優先度**: Must
- **ステータス**: Draft
- **説明**: 「位置（および任意の入力属性）を受け取り値を返す関数」としての**フィールド**を、特定機能専用ではない共通インターフェースで定義する。ノイズ、フォールオフ（球/線形/パス距離）、カーブリマップ、画像サンプラ、式（Lua）、オーディオ由来の値等をフィールドとして提供し、任意のジオメトリ属性の変調・パーティクルフォース・per-instance パラメータ変調（REQ-MOGRAPH-001）に同一の仕組みで接続できる。フィールドはノードグラフ上の接続として合成（加算・乗算・最大等）可能。
- **受入条件**:
  - [ ] フィールドがノード間を流れる型として存在する
  - [ ] ノイズ/フォールオフ/カーブ/画像/式のビルトインフィールドがある
  - [ ] 任意のジオメトリ属性をフィールドで変調できる
  - [ ] フィールド同士をブレンド・合成できる
  - [ ] パーティクルフォース（REQ-MOGRAPH-002）がフィールドとして実装される
  - [ ] 統一アニメーションチャネル（REQ-CORE-007)の値ソースとして接続できる
- **依存**: REQ-CORE-003, REQ-CORE-010

## REQ-CORE-013: グラフ内反復

- **優先度**: Could
- **ステータス**: Draft
- **説明**: Houdini for-each / Geometry Nodes repeat zone 相当のグラフ内ループ。初期リリースでは**採用しない**ことを明示する: 配列一括演算（属性システム + フィールドによる全要素同時変調）で主要ユースケースを代替し、評価エンジンは静的 DAG 前提を維持する。要素ごとの分岐が必要なケースはサブグラフ + per-instance 属性で表現する。将来採用する場合は評価エンジンの展開（unroll）方式を別途設計する。
- **受入条件**:
  - [ ] 配列一括演算 + 属性で反復なしに代表ユースケース（複製変調、文字単位アニメーション）が実現できることを確認する
  - [ ] 採用判断の再評価タイミング（MOGRAPH 実装後）が記録されている
- **依存**: REQ-CORE-010, REQ-CORE-012
